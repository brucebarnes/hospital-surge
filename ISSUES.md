# Hospital-Surge: Issue Backlog

Generated by code analysis of `src/App.jsx`. Issues are grouped by priority.

## 🔴 CRITICAL BUGS (Fix immediately — core features broken)

### BUG-1: Partner Hospital transfer counter never increments
**File:** `src/App.jsx:670`
**Label:** `bug`, `priority:critical`

The `totPartner` counter adds zero instead of one, so Partner Hospital transfers always report as 0.

```js
// Current (broken)
cur.totPartner = (cur.totPartner ?? 0) + 0;

// Fix
cur.totPartner = (cur.totPartner ?? 0) + 1;
```

**Impact:** After-action reports always show 0 Partner Hospital transfers regardless of simulation events.

---

### BUG-2: `wardDivertChoice` ignored — all diverted patients route to ORC regardless of modal selection
**File:** `src/App.jsx:107–112, 225–229, 666`
**Label:** `bug`, `priority:critical`

The Ward diversion modal correctly captures the commander's choice (ORC vs. Partner Hospital), but the routing logic in `tick()` never reads `wardDivertChoice`. All diverted patients always go to ORC.

**Fix:** Read `wardDivertChoice` (or a ref) in the Ward overflow branches and route accordingly:
```js
const dest = polRef.current.wardDivertChoice === 'Partner' ? 'partner' : 'orc';
p.cur = dest;
```

**Impact:** Selecting "Partner Hospital" has zero effect on the simulation. The decision UI is completely non-functional.

---

### BUG-3: `wardDivertCap` is never enforced
**File:** `src/App.jsx:50–52, 101, 467, 591`
**Label:** `bug`, `priority:critical`

`wardDivertCap` (default 20 patients/day) is defined, displayed in the policy panel, and referenced in comments — but there is no daily counter and no logic to block admissions once the cap is reached.

**Fix:**
1. Add a `dailyDivertCount` field to simulation state, reset to 0 at each new sim-day.
2. Before diverting a patient, check `dailyDivertCount < wardDivertCap`; if over cap, reject/queue the patient.

**Impact:** Unlimited patients can be diverted per day; the configurable cap is a UI-only placeholder.

---

### BUG-4: Partner Hospital has no simulation model (infinite capacity, no census)
**File:** `src/App.jsx:21–32, 150–152`
**Label:** `bug`, `priority:critical`

Partner Hospital is referenced in the UI and diversion logic but is **not** a simulated unit. It has no bed array, no occupancy tracking, and no capacity constraint. `partnerCapacity` is purely a display annotation.

**Fix:** Add Partner Hospital to the units array with licensed beds = `partnerCapacity`, surge = 0, and census tracking. Block new admissions when at capacity (or model a rejection/queue).

**Impact:** Routing patients to Partner Hospital has no simulation consequences — unlimited overflow is silently absorbed.

## 🟠 HIGH PRIORITY (Fix before next exercise)

### BUG-5: Simulation seed not reproducible across resets
**File:** `src/App.jsx:148, 655`
**Label:** `bug`, `priority:high`

Initial load uses seed 42 (hardcoded). The Reset button generates a **random** seed via `Math.random()`, breaking the ability to reproduce results. The UI allows a seed to be set but resets it randomly on each restart.

**Fix:** On Reset, use the currently configured `pol.seed` value instead of generating a new random one.

---

### PERF-1: `JSON.parse(JSON.stringify(...))` deep copy inside animation loop
**File:** `src/App.jsx:688`
**Label:** `performance`, `priority:high`

A full deep clone of the units array runs every animation frame (~60 fps). This creates significant GC pressure and will cause visible lag on longer simulations.

**Fix:** Use structured clone, Immer draft, or a targeted shallow copy of only mutated fields.

---

### QUALITY-1: No input validation — negative/zero values accepted silently
**File:** `src/App.jsx:492, 506`
**Label:** `code-quality`, `priority:high`

Numeric inputs fall back to 0 on invalid values (`parseFloat(e.target.value)||0`). There is no guard against:
- Negative bed counts
- Zero LOS (causes infinite loops in LOS sampling)
- Acuity fractions summing > 1
- Division-by-zero in bed ratio calculations (patched with `Math.max(...,1)` workarounds throughout)

**Fix:** Add min/max validation at input boundaries; show inline error messages instead of silent coercion.

---

### QUALITY-2: No test coverage
**Label:** `testing`, `priority:high`

There are zero tests. The simulation contains non-trivial stochastic logic (`tick()`, `admitPt()`, patient routing, LOS sampling) with no regression safety net.

**Suggested tests:**
- Unit: Poisson arrival sampling, log-normal LOS sampling
- Unit: Acuity routing paths (critical/high/moderate/low)
- Unit: Surge/diversion threshold triggers
- Unit: Air Chalk (CHALK) batch arrival routing
- Integration: Ward diversion → ORC and → Partner Hospital paths
- Integration: Supply depletion and restocking cycle

## 🟡 MEDIUM PRIORITY (Polish and maintainability)

### QUALITY-3: Entire app is a single 889-line JSX file with no component decomposition
**File:** `src/App.jsx`
**Label:** `refactor`, `priority:medium`

All simulation logic, UI components, and state management are intermingled in one file. This makes debugging, testing, and onboarding very difficult.

**Suggested decomposition:**
- `src/sim/engine.js` — `tick()`, `admitPt()`, `createState()`
- `src/sim/rng.js` — seeded RNG, Poisson/log-normal samplers
- `src/components/BedGrid.jsx` — bed occupancy visualization
- `src/components/CensusChart.jsx` — Recharts wrapper
- `src/components/ConfigPanel.jsx` — unit/policy configuration
- `src/components/SupplyPanel.jsx` — supply chain controls

---

### QUALITY-4: Magic numbers and hardcoded constants throughout
**File:** `src/App.jsx:308, 685, 703, 747`
**Label:** `code-quality`, `priority:medium`

Examples:
- `20` — bed grid max columns (`Math.min(..., 20)`)
- `0.2` — animation frame delta cap in seconds
- `300` — max chart data points
- `245px` — left panel width
- Colors defined as single-letter keys (`S.r`, `S.g`, etc.)

**Fix:** Extract to named constants at the top of the file (or a `constants.js` module).

---

### QUALITY-5: Chart data is lost after ~5 simulated days
**File:** `src/App.jsx:703`
**Label:** `bug`, `priority:medium`

Chart history is capped at 300 points and uses `0.04-day` (~57-minute) throttling. At 1-hour resolution, only the last ~12.5 days of data are retained. The throttle check (`Math.abs(last.day - next.t/24) < 0.04`) is also fragile.

**Fix:** Separate raw simulation data from chart render data; down-sample for display while retaining full history for export.

---

### QUALITY-6: Inconsistent null/undefined guards
**File:** `src/App.jsx` throughout
**Label:** `code-quality`, `priority:medium`

Mixed patterns for null checks: early returns, optional chaining, nested conditionals, and `if(!p)return` vs `if(p)` after `.find()`. Makes control flow hard to trace.

**Fix:** Standardize on optional chaining + nullish coalescing with early returns in simulation functions.
